# 実装計画：ASTおよびTokenのリファクタリングと動的メモリ確保

## 目的
`parser.h` で定義されている `node_t` と `token_t` を整理し、これまでの `MAX_STMTS` (最大100) や `MAX_ARGS` (最大8) といった静的配列による固定サイズ制限を撤廃して、`malloc`/`realloc` を用いた動的な配列割り当て (動的メモリ確保) へと移行する。

## 実装ステップ

### 1. `parser.h` のデータ構造改善
- `node_t` 内の乱雑に配置されていたメンバ変数を「ツリー構造用」「関数/ブロック用」「制御構造用」「データ型/リテラル用」といった用途ごとに整理して定義。
- `MAX_STMTS` および `MAX_ARGS` のマクロ定義を削除。
- `node_t *body[MAX_STMTS]` を `node_t **body` (ポインタの配列へのポインタ) に変更。
- `node_t *args[MAX_ARGS]` を `node_t **args` に変更。
- `extern node_t *code[MAX_STMTS]` を `extern node_t **code` に変更し、プログラムトップレベルのリストも同様に動的割り当て管理とする。

### 2. `parser.c` のパース処理とデータ割り当ての改修
- トップレベルのプログラム処理 (`program`) において、`code` 配列の初期キャパシティ (例: 8) を `calloc` で確保し、閾値に達した場合は `realloc` を用いて動的に拡張するロジックを追加。
- ブロック文 (`{ ... }`) のパース時に `body` の配列表現を同様の `calloc` と `realloc` の仕組みに置き換え、各文のパースを続ける。
- 関数定義および関数呼び出しにおける実引数/仮引数の処理 (`node->args`) についても同様に `realloc` を使用して拡張する。
- 終端には必ず `NULL` ポインタを設定し、テスト・コード生成時に安全に要素数を判定またはループを終了できるようにする。

### 3. 未初期化アクセス・クラッシュ問題の解決
- `test_codegen.c` などの、ASTを手動で作成してコード生成をテストしているテストファイルにおいて、ASTの `body` および `args` に手動でノードを代入する直前に `calloc` で配列ポインタを確保するように修正。
- ASan (AddressSanitizer) などのメモリ検出ツールを用いてテストを実行し、メモリ破壊やセグメンテーション違反が発生しないことを確証する。

### 4. 単体テストおよびE2Eテストの実行
- トークナイザ、パーサー、コード生成、E2Eテスト、プリプロセッサ全般のテストスイートを通しで実行し、既存のコード生成処理に影響・後退がないかを検証する。
- 全てのテストが成功したら完了とする。
